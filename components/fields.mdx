## Overview

Field components are specialized MDX components for documenting API parameters, response structures, and HTTP responses. These components provide consistent, professional formatting for technical specifications.

## ParameterField

Document request parameters, headers, query strings, and body fields with clear type information and validation indicators.

### Basic Usage

<TabGroup>
  <Tab title="Preview">
  <ParameterField name="userId" type="string" required>
  Unique identifier for the user account.
  </ParameterField>
  </Tab>

  <Tab title="Syntax">
  ```markdown
  <ParameterField name="userId" type="string" required>
  Unique identifier for the user account.
  </ParameterField>
  ```
  </Tab>
</TabGroup>

### Available Props

<ParameterField name="name" type="string" required>
The parameter name. Displayed in monospace font with brand color.
</ParameterField>

<ParameterField name="type" type="string" required>
Data type of the parameter (e.g., `string`, `integer`, `boolean`, `array`, `object`).
</ParameterField>

<ParameterField name="required" type="boolean" default="false">
Displays a red "required" badge when set to `true`.
</ParameterField>

<ParameterField name="default" type="string">
Default value for the parameter. Displays as a badge with the format `default: value`.
</ParameterField>

<ParameterField name="deprecated" type="boolean" default="false">
Displays a yellow "deprecated" badge when set to `true`. Use for parameters being phased out.
</ParameterField>

<ParameterField name="example" type="string">
Example value for the parameter. Displays as a badge with the format `example: value`.
</ParameterField>

<ParameterField name="enum" type="string[]">
Array of allowed values. Displays as comma-separated list in a badge.
</ParameterField>

<ParameterField name="children" type="ReactNode">
Description text and rich content. Supports full MDX formatting including code blocks, lists, and nested components.
</ParameterField>

## APIResponse & APIResponseField

Document HTTP responses with status codes, descriptions, and field schemas using collapsible response containers.

### Basic Usage

<TabGroup>
  <Tab title="Preview">
  <APIResponse status="200" description="Successful response" contentType="application/json" defaultOpen>
    <APIResponseField name="id" type="string" description="Resource identifier" />
    <APIResponseField name="name" type="string" description="Resource name" />
  </APIResponse>
  </Tab>

  <Tab title="Syntax">
  ```markdown
  <APIResponse status="200" description="Successful response" contentType="application/json" defaultOpen>
    <APIResponseField name="id" type="string" description="Resource identifier" />
    <APIResponseField name="name" type="string" description="Resource name" />
  </APIResponse>
  ```
  </Tab>
</TabGroup>

### APIResponse Props

<ParameterField name="status" type="string" required>
HTTP status code (e.g., `"200"`, `"400"`, `"500"`). Determines color-coding and status text.
</ParameterField>

<ParameterField name="description" type="string">
Human-readable description of the response.
</ParameterField>

<ParameterField name="contentType" type="string">
MIME type of the response (e.g., `"application/json"`, `"text/plain"`). Displayed on larger screens.
</ParameterField>

<ParameterField name="defaultOpen" type="boolean" default="false">
Whether the response accordion starts in the open state. Set to `true` for primary success responses.
</ParameterField>

<ParameterField name="children" type="ReactNode">
`APIResponseField` components describing the response schema.
</ParameterField>

### APIResponseField Props

<ParameterField name="name" type="string" required>
Field name in the response.
</ParameterField>

<ParameterField name="type" type="string" required>
Data type of the field.
</ParameterField>

<ParameterField name="description" type="string">
Brief description of the field's purpose.
</ParameterField>

### Status Code Color System

The component automatically color-codes responses based on status code ranges:

- **2xx (Green)**: Success responses - emerald color scheme
- **3xx (Blue)**: Redirect responses - blue color scheme  
- **4xx (Amber)**: Client error responses - amber color scheme
- **5xx (Red)**: Server error responses - red color scheme

### Supported Status Codes

The component includes human-readable text for common status codes:

<Expand title="Success Responses (2xx)">
- **200**: OK
- **201**: Created
- **202**: Accepted
- **204**: No Content
</Expand>

<Expand title="Redirect Responses (3xx)">
- **301**: Moved Permanently
- **302**: Found
- **304**: Not Modified
</Expand>

<Expand title="Client Error Responses (4xx)">
- **400**: Bad Request
- **401**: Unauthorized
- **403**: Forbidden
- **404**: Not Found
- **405**: Method Not Allowed
- **409**: Conflict
- **422**: Unprocessable Entity
- **429**: Too Many Requests
</Expand>

<Expand title="Server Error Responses (5xx)">
- **500**: Internal Server Error
- **502**: Bad Gateway
- **503**: Service Unavailable
</Expand>

## Best Practices

### Component Selection

- Use **ParameterField** for request parameters (headers, query strings, path parameters, request body)
- Use **ResponseField** for detailed response object documentation with nested structures
- Use **APIResponse** with **APIResponseField** for HTTP response schemas with status codes

### Naming Conventions

- Use exact field names as they appear in your API (case-sensitive)
- Match parameter names to your API specification precisely
- Use descriptive, consistent naming across all endpoints

### Required vs Optional

- Always mark required fields with `required` prop
- Document default values for optional parameters
- Explain the behavior when optional fields are omitted

### Type Specification

Common type values:
- Primitives: `string`, `integer`, `number`, `boolean`
- Complex: `object`, `array`
- Special: `timestamp`, `uuid`, `email`, `url`

### Examples and Enums

- Provide realistic examples that users can copy
- Use enum for fields with limited allowed values
- Show example values in the exact format expected by the API

### Descriptions

- Write concise, action-oriented descriptions
- Include validation rules and constraints
- Mention format requirements (ISO 8601 for dates, UUID format, etc.)
- Link to related documentation when helpful

### Response Documentation

- Set `defaultOpen={true}` on success responses (2xx)
- Document all possible error responses
- Include error field schemas for 4xx and 5xx responses
- Specify content type for all responses

### Organization

- Group related parameters with Expand components
- Organize parameters by type: headers, path, query, body
- Present responses in order: success first, then errors by status code
- Use consistent section headings across all endpoints

### Accessibility

- Provide clear, descriptive text for all fields
- Use semantic HTML structure with proper headings
- Ensure color-coded badges have text labels (not color-only indicators)
