## Overview

Devscribe's AI-powered documentation generation allows you to connect a code repository and automatically create comprehensive documentation. The system analyzes your code, understands its structure, and produces formatted MDX documentation that's immediately ready for publishing.

<Callout type="info">
Generated documentation is automatically added to your `docs.json` file, making it appear in your documentation site immediately after saving.
</Callout>

## How Doc Generation Works

The documentation generation system follows a three-step workflow:

<Steps>
<Step title="Select documentation type">
Choose the type of documentation you want to generate:

- **API Reference** - Complete endpoint documentation with schemas, authentication, and code examples
- **External Docs** - User-facing guides, tutorials, and feature documentation  
- **Internal Docs** - Technical architecture and implementation documentation
</Step>

<Step title="Configure repository">
Connect your code repository:

1. Select a repository from your connected GitHub or GitLab account
2. Choose a specific branch (optional - defaults to main/master)
3. For API documentation: specify the endpoint URL you want to document

<Callout type="tip">
You need to authenticate with GitHub or GitLab before repositories will appear in the selection dropdown.
</Callout>
</Step>

<Step title="Generate and refine">
Use the generation interface:

- **Left sidebar** - View and manage generated documents with save status tracking
- **Center panel** - Edit generated content with Preview/Edit toggle
- **Right sidebar** - Select context files and enter generation prompts
</Step>
</Steps>

## Connecting a Repository

### Authentication and Access

Before generating documentation, you must connect your code hosting provider:

1. Navigate to the generation interface
2. Click the repository dropdown
3. If not authenticated, you'll be prompted to connect GitHub or GitLab
4. Authorize Devscribe to access your repositories

<Callout type="info">
Devscribe only requests access to repositories where you've installed the GitHub App or granted GitLab permissions.
</Callout>

### Repository Selection Process

Once authenticated, the system:

1. Fetches all accessible repositories using your `installation_id`
2. Displays repositories with owner, name, and last update information
3. Loads available branches when you select a repository
4. Retrieves the complete file tree for context selection

<CodeSnippet language="typescript" title="Repository data structure">
{`interface GitHubRepo {
  id: number;
  name: string;
  full_name: string;
  owner: { login: string };
  description?: string;
  updated_at: string;
}`}
</CodeSnippet>

## How the Agent Parses Your Repository

### Context Selection

The AI agent requires context to understand your codebase. You can select up to 5 files or directory paths:

1. Browse your repository's file tree in the right sidebar
2. Click files or folders to add them as context
3. Selected paths guide the agent's analysis and generation

<Callout type="tip">
Choose relevant files that contain the core logic for what you're documenting. For API docs, select route handlers, controllers, or endpoint definitions.
</Callout>

### Generation Request

When you submit a generation prompt, the system sends:

<ParameterField name="prompt" type="string" required>
Your natural language instructions describing what documentation to create
</ParameterField>

<ParameterField name="repo" type="string" required>
Repository identifier in `owner/repo` format
</ParameterField>

<ParameterField name="branch" type="string">
Specific branch name, or empty for default branch
</ParameterField>

<ParameterField name="paths" type="array" required>
Up to 5 file or directory paths selected as context
</ParameterField>

<ParameterField name="docType" type="string" required>
Documentation type: `"API References"`, `"External Docs"`, or `"Internal Docs"`
</ParameterField>

<ParameterField name="endpoint" type="string">
For API documentation only - the endpoint URL to document
</ParameterField>

### Streaming Response

The agent generates documentation in two phases:

1. **Analysis phase** - Agent "thinks" through the code structure and documentation approach
2. **Content generation** - Formatted MDX documentation streams to your editor

<Callout type="success">
Documentation appears in real-time as the agent generates it, allowing you to see progress immediately.
</Callout>

### Iterative Refinement

After initial generation, you can refine the documentation:

- Enter follow-up prompts to adjust content
- Chat history is maintained for context
- Agent understands previous generations and edits accordingly

## Adding Documentation to docs.json

### Automatic Addition

When you save generated documentation, the system **automatically updates your docs.json file**:

1. Extracts the `title` from the document's frontmatter
2. Creates a slugified filename (e.g., "User Authentication" becomes `user-authentication`)
3. Determines the appropriate tab based on document type
4. Adds the page entry to docs.json
5. Saves both the document and updated docs.json to storage

<Callout type="warning">
Generated documentation is saved to the `generated/` directory. The page entry in docs.json uses the format `"generated/{slug}"` without the `.mdx` extension.
</Callout>

### docs.json Structure

Your docs.json file controls all documentation site navigation:

<CodeSnippet language="json" title="docs.json">
{`{
  "name": "your-project",
  "favicon": "/favicon.svg",
  "navigation": {
    "tabs": [
      {
        "tab": "Guides",
        "groups": [
          {
            "group": "Get Started",
            "pages": ["introduction", "quickstart"]
          },
          {
            "group": "Generated Documentation",
            "pages": [
              "generated/user-authentication",
              "generated/api-overview"
            ]
          }
        ]
      },
      {
        "tab": "API Reference",
        "groups": [
          {
            "group": "Endpoints",
            "pages": [
              "generated/create-user",
              "generated/get-user"
            ]
          }
        ]
      }
    ]
  }
}`}
</CodeSnippet>

### Tab Assignment Logic

Generated documentation is added to tabs based on type:

- **API Reference docs** - Added to the "API Reference" tab if it exists, otherwise the first tab
- **External/Internal docs** - Added to the first tab in your navigation

<Callout type="info">
If an API Reference tab doesn't exist when generating API docs, the system adds the page to the first available tab. You can manually move the page entry to a different tab by editing docs.json.
</Callout>

### Manual docs.json Management

You can manually edit docs.json to:

- Reorganize page order within groups
- Move pages between groups or tabs
- Create new groups for better organization
- Add icons to groups or pages

<Callout type="danger">
docs.json cannot be deleted, moved, or renamed. Any MDX files not referenced in docs.json will not appear on your documentation site.
</Callout>

<CodeSnippet language="json" title="Page entry format">
{`{
  "group": "Authentication",
  "pages": [
    "generated/oauth-setup",
    "generated/api-keys"
  ]
}`}
</CodeSnippet>

## Best Practices

<AccordionGroup>
<Accordion title="Plan before generating">
Determine exactly what should be documented before writing your prompt. Clear goals produce focused, useful documentation.
</Accordion>

<Accordion title="Write specific prompts">
Detailed instructions yield better results. Instead of "document this API", try "create a comprehensive guide for the user creation endpoint including authentication, request/response examples, and error handling".
</Accordion>

<Accordion title="Select targeted context">
Choose the 5 most relevant files that contain the core logic you're documenting. More context isn't always better - focus on quality over quantity.
</Accordion>

<Accordion title="Review and refine">
Treat generated documentation as a first draft. Review for accuracy, add domain-specific details, and refine examples to match your style.
</Accordion>

<Accordion title="Organize with groups">
As you generate more documentation, create logical groups in docs.json to maintain clear navigation structure.
</Accordion>
</AccordionGroup>

## Storage and Deployment

### File Structure

Generated documentation is stored in Google Cloud Storage with this structure:

<CodeSnippet language="plaintext">
{platform_id}/
├── docs/
│   ├── docs.json
│   └── generated/
│       ├── user-authentication.mdx
│       ├── api-overview.mdx
│       └── endpoint-reference.mdx
└── metadata.json
</CodeSnippet>

### Deployment Status

Each file is tracked with deployment status in `metadata.json`:

<ResponseField name="isDeployed" type="boolean">
Whether the file has been published to your live documentation site
</ResponseField>

<ResponseField name="originalHash" type="string">
Content hash when file was last deployed
</ResponseField>

<ResponseField name="currentHash" type="string">
Content hash of current file version
</ResponseField>

<Callout type="info">
When `originalHash` and `currentHash` differ, the file has unpublished changes that require deployment.
</Callout>

## Troubleshooting

<AccordionGroup>
<Accordion title="Repository not appearing in dropdown">
- Verify you've authenticated with GitHub or GitLab
- Check that you've installed the GitHub App or granted GitLab permissions
- Ensure the repository is accessible with your current account
</Accordion>

<Accordion title="Generated doc not showing on site">
- Confirm the page entry was added to docs.json
- Check that the page path matches the generated file location
- Verify the file path uses the correct format: `"generated/{slug}"` without `.mdx`
- Deploy your documentation to publish changes
</Accordion>

<Accordion title="Agent generation fails or times out">
- Reduce the number of context files selected
- Choose smaller, more focused files rather than large directories
- Break complex documentation into multiple smaller generations
- Check that your repository is accessible and the branch exists
</Accordion>

<Accordion title="Context files not loading">
- Wait for the file tree to fully load after selecting a repository
- Try selecting a different branch
- Verify the repository contains files (not an empty repo)
</Accordion>
</AccordionGroup>
